# E6: PR Review and Coding Standards

This document is the common PR review standard for PawIt repositories. Use it after every PR is opened. A PR should be merged only after the E6 review checklist is satisfied, CI is passing, and the reviewer is confident the change is production-ready.

## Review Gate

- Every PR must have a clear title, short summary, and testing notes.
- Every PR must explain user-facing behavior changes, API changes, database changes, or deployment changes when applicable.
- CI, linting, formatting, unit tests, and contract tests must pass before merge.
- Risky changes must include focused tests for the behavior being changed.
- Database migrations must be reversible or clearly documented if rollback is not possible.
- API changes must match the shared OpenAPI contract and must not silently break frontend clients.
- Security-sensitive changes must be reviewed for authentication, authorization, tenant isolation, secrets, logging, and data exposure.
- The reviewer must block merge when correctness, security, data safety, or maintainability concerns remain unresolved.

## Coding Standards

- Follow the existing patterns of the repository before introducing new abstractions.
- Keep changes focused on the PR goal. Avoid unrelated refactors, formatting churn, and broad rewrites.
- Prefer simple, readable code over clever code.
- Use meaningful names for packages, functions, variables, request fields, and response fields.
- Keep business rules in domain or service layers instead of scattering them across handlers or UI components.
- Validate inputs at system boundaries and return consistent error responses.
- Preserve multi-tenant safety by always scoping data access by tenant where applicable.
- Make authorization checks explicit for role-based or permission-based workflows.
- Handle errors intentionally. Do not swallow errors, panic in request paths, or leak internal details to clients.
- Add tests for new behavior, bug fixes, permissions, validation, and important edge cases.
- Keep documentation updated when endpoints, workflows, environment variables, migrations, or deployment steps change.
- Do not commit secrets, credentials, tokens, private keys, production data, or local-only files.

## Senior Engineer PR Review

A senior engineer reviews the PR from multiple angles, not just whether the code compiles.

- Understand the intent: confirm the PR solves the right problem and the scope matches the stated goal.
- Check correctness: trace the main path, failure paths, edge cases, and data state transitions.
- Check system impact: look for broken contracts, migration risks, concurrency issues, performance regressions, and deployment surprises.
- Check security: verify auth, RBAC, tenant isolation, input validation, secret handling, audit logging, and sensitive data exposure.
- Check maintainability: ensure the design fits the codebase, keeps responsibilities clear, and avoids unnecessary complexity.
- Check test quality: tests should prove behavior, not just execute lines. They should cover meaningful success and failure cases.
- Check observability: important failures should be diagnosable through logs, errors, metrics, or audit records without exposing private data.
- Check developer experience: naming, structure, and documentation should make the next change easier to make safely.

## Merge Checklist

Before merging, confirm:

- The PR branch is up to date with the target branch or conflicts are resolved.
- The diff is focused and understandable.
- All requested changes are addressed.
- CI and required checks are green.
- Tests were added or the lack of new tests is justified.
- API, database, deployment, and documentation changes are reflected in the right files.
- The reviewer has completed the E6 review and explicitly approved the PR.

If any item is not satisfied, do not merge yet. Request changes, add the missing work, and re-run the review.
